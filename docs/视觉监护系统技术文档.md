## 3. 讯飞 M260C USB音频配置

> M260C 是 USB Audio Class 设备，Linux 内核自带驱动，**不需要安装任何额外驱动**。  
> 插上就识别为 USB 声卡，用 ALSA 的 `aplay` 直接播放 WAV 文件。

### 3.1 插上后确认设备识别

```bash
# 步骤1: USB设备层面确认
lsusb
# 正常输出示例:
#   Bus 001 Device 003: ID xxxx:xxxx Generic USB Audio
#   或显示为 "iFlytek" / "M260C" / "USB Advanced Audio Device"
#   只要出现 "Audio" 字样就是识别成功

# 步骤2: ALSA声卡层面确认
cat /proc/asound/cards
# 正常输出示例:
#    0 [x5audio       ]: x5-audio - x5-audio
#                       地平线 x5-audio
#    1 [UAC2Demo      ]: USB-Audio - UAC2Demo
#                        Generic USB Audio
#                         ↑ card 1 就是 M260C

# 步骤3: 列出所有PCM播放设备
aplay -l
# 正常输出示例:
#   **** List of PLAYBACK Hardware Devices ****
#   card 0: x5audio [x5-audio], device 0: ...  ← RDK X5 板载声卡
#     Subdevices: 1/1
#     Subdevice #0: subdevice #0
#   card 1: UAC2Demo [UAC2Demo], device 0: USB Audio  ← ★ M260C
#     Subdevices: 1/1
#     Subdevice #0: subdevice #0
```

### 3.2 理解 `aplay -l` 输出 → 配置 `audio_device`

| `aplay -l` 输出 | ALSA 设备名 | 含义 | 使用场景 |
|-----------------|-------------|------|----------|
| card 0, device 0 | `plughw:0,0` | 板载声卡 | HDMI/3.5mm输出 |
| **card 1, device 0** | **`plughw:1,0`** | **M260C (第一个USB声卡)** | **通常是这个** |
| card 2, device 0 | `plughw:2,0` | M260C (第二个USB声卡) | 如果板载也占用了一个 |
| — | `default` | 系统默认设备 | 不指定设备时的输出 |

> **关键**: `plughw:1,0` 中的 `1` = card 编号，`0` = device 编号。  
> 看你的 `aplay -l` 输出中 M260C 是 card 几，就写 `plughw:几,0`。

### 3.3 三种场景的 `audio_device` 配置

```yaml
# 场景A: M260C 是系统唯一USB声卡 (最常见)
# aplay -l 显示 card 1 = UAC2Demo
audio_device: "plughw:1,0"

# 场景B: 不确定哪个是M260C, 用 default 碰运气
audio_device: "default"

# 场景C: 多个USB声卡, 按名字精确匹配
audio_device: "plughw:CARD=UAC2Demo,DEV=0"
#          或 "plughw:CARD=M260C,DEV=0"
#          CARD= 后面的名字来自 /proc/asound/cards 中方括号里的名字
```

### 3.4 手动测试 M260C 能否出声

```bash
# 方法1: 生成 + 播放测试音 (一键)
python3 -c "
import wave, struct, math
sr, dur = 44100, 1.5
n = int(sr * dur)
with wave.open('/tmp/test_beep.wav', 'w') as w:
    w.setnchannels(1); w.setsampwidth(2); w.setframerate(sr)
    data = b''
    for i in range(n):
        t = i / sr
        f = 440 + 440 * t / dur  # 440→880Hz 上升
        v = int(20000 * math.sin(2*math.pi*f*t) * math.exp(-t*2))
        data += struct.pack('<h', max(-32767, min(32767, v)))
    w.writeframes(data)
"

# 方法2: 分别用不同设备播放, 哪个出声就知道了
aplay -D plughw:1,0 /tmp/test_beep.wav   # 尝试 card 1
aplay -D plughw:2,0 /tmp/test_beep.wav   # 尝试 card 2
aplay -D default /tmp/test_beep.wav       # 尝试默认

# 听到声音 → 那个设备就是对的 → 记下来填入 vision_params.yaml
```

### 3.5 设置默认音频设备 (可选, 一劳永逸)

```bash
# 如果想让 M260C 成为系统默认音频设备, 创建 /etc/asound.conf:
sudo tee /etc/asound.conf << 'EOF'
pcm.!default {
    type hw
    card 1          # ← 改成你的 M260C card 编号
    device 0
}
ctl.!default {
    type hw
    card 1
}
EOF

# 然后 audio_device 就可以直接用 "default" 了
```

### 3.6 M260C 常见问题

```bash
# 问题1: lsusb 看不到设备
# → 换一个USB口, 换一条USB线, 检查M260C指示灯是否亮

# 问题2: aplay -l 能看到但没声音
# → 检查M260C音量旋钮是否拧到最小
# → 用 amixer 检查静音状态:
amixer -c 1 scontents    # 查看 card 1 的混音器状态

# 问题3: aplay 报错 "Device or resource busy"
# → 有其他程序占用了声卡, 杀掉它们:
sudo fuser /dev/snd/*    # 查看哪个进程占用
sudo kill -9 <PID>

# 问题4: 播放声音很小
# → M260C 自带功放, 确认USB供电充足
# → 修改声音代码中的 AMPLITUDE 值 (AlarmSoundGenerator.AMPLITUDE = 16000)
```

## 4. LED灯光报警接线

```
RDK X5 40Pin 排针 (BCM编号):

                   3.3V  (pin 1)      (pin 2)  5V
                   GPIO2 (pin 3)      (pin 4)  5V
                   GPIO3 (pin 5)      (pin 6)  GND
                   ...   ...
                  GPIO23 (pin 16)  ────→ [220Ω]→[LED+]→[LED-]→ GND (pin 14)
                  GPIO24 (pin 18)     (pin 20) GND
                   ...   ...
                                     (pin 39) GND
                                     (pin 40) GPIO21

接线细节:
  GPIO23 ──→ [220Ω电阻] ──→ LED长脚(正极/Anode)
                            LED短脚(负极/Cathode) ──→ GND

辨认LED极性:
  长脚 = 正极(+)
  短脚/平边 = 负极(-)
  灯珠内部: 小金属片 = 正极, 大金属片(碗状) = 负极

为什么串220Ω?
  GPIO输出3.3V, LED正向压降≈2V
  电流 = (3.3-2.0)/220 = 5.9mA (安全范围, 亮度足够)
  不串电阻 → 电流过大 → 烧LED或GPIO口
```

## 6. 核心算法说明

### 6.1 3D人体反投影

```
输入: 2D bbox中心 (u,v) + 深度Z (来自双目对齐深度图)
输出: 3D相机坐标 (X, Y, Z)

针孔相机模型:
    X = (u - cx) × Z / fx
    Y = (v - cy) × Z / fy
    Z = depth(u, v)

世界离地高度:
    H_ground = camera_height - Y_world

其中 Y_world 需考虑摄像头俯仰角 pitch:
    Y_world = Y_cam × cos(pitch) + Z_cam × sin(pitch)
```

### 6.2 卡尔曼滤波去噪

双目深度在边缘区域(头发/黑色衣物)噪声较大。一维卡尔曼滤波平滑:

```
预测:  P = P + Q
更新:  K = P / (P + R)
       state = state + K × (measurement - state)
       P = P × (1 - K)

Q=0.01 (过程噪声)  R=0.05 (测量噪声)
```

### 6.3 多条件融合跌倒判决

| 条件 | 阈值 | 权重 | 说明 |
|------|------|------|------|
| 头部离地高度 | <0.55m | 高 | 主要判据 |
| 人体宽高比 | >1.15 | 中 | 站立≈0.3-0.5, 躺倒≈1.5-2.0 |
| 持续时长 | >1.5s | 高 | 过滤瞬间弯腰/蹲下 |

判决逻辑:
```
高度条件 + 宽高比条件 → fast accumulate (×1.0)
仅高度条件           → slow accumulate (×0.5)
不满足               → decay (-0.5/帧)
fall_counter >= 15   → trigger alarm (1.5s × 10Hz)
```

### 6.4 火焰检测 (双模态)

| 模式 | 方法 | 说明 |
|------|------|------|
| BPU检测 (优先) | YOLOv5s-fire 模型 | 地平线Model Zoo |
| HSV颜色检测 (后备) | 橙红-黄 HSV + 连通域 | CPU实时, 零依赖 |

### 7.2 跌倒场景细节

```
测试1: 正面跌倒 → 面对机器人慢慢坐到地上躺平
测试2: 侧面跌倒 → 侧身倒地
测试3: 椅子滑落 → 从椅子缓慢滑到地面 (最接近老人真实跌倒)
测试4: 遮挡跌倒 → 下半身被椅子挡住, 仅头部可见

每次跌倒后观察:
  ① M260C 播放低沉警笛声 (fall_alarm.wav)
  ② GPIO23 LED 红色闪烁
  ③ 机器人自动停车
  ④ 终端红色报警日志
  ⑤ 自动拍照存证 → ~/snapshots/fall/
```

### 7.3 视频制作建议

```
剪辑:
  前10秒快剪: 报警瞬间 + LED闪烁 + 警笛声
  主体: 按场景顺序, 每场景加文字标题
  对比: 正常行为 vs 跌倒检测 (并排对比)
  强调: 报警瞬间加红色边框 + 音频波形图

  摇晃镜头 → 用三脚架

## 8. 答辩技术亮点

| 创新点 | 技术内容 | 关键词 |
|--------|----------|--------|
| **3D空间理解** | bbox + 深度反投影 → 3D头部位置 | "不是简单2D分类" |
| **卡尔曼滤波** | 抑制双目深度噪声, 平滑高度估计 | "时序状态估计" |
| **多条件融合** | 高度+宽高比+时序, 区分蹲下和跌倒 | "降低误报率" |
| **BPU推理加速** | NPU ~10ms vs CPU ~200ms | "边缘AI实时推理" |
| **RGB-D对齐** | 双目硬件对齐, 无需手动标定 | "传感器融合" |
| **主动响应** | 报警→停车→声光→拍照存证 | "感知-决策-执行闭环" |
| **USB音频报警** | M260C声卡 → 程序生成WAV → aplay播放 | "零外部依赖, 自适应音效" |

### 预期Q&A

**Q: 为什么不用纯RGB?**
A: 纯RGB无3D位置。蹲在近处和躺在远处在2D上相似。深度信息解决尺度二义性。

**Q: 蹲下会不会误报?**
A: 不会。蹲下宽高比仍<1.0, 且不持续1.5s以上。多条件融合机制专门区分。

**Q: 深度噪声怎么处理?**
A: 三层: ROI中值采样(滤离群噪点) → 卡尔曼时序平滑(滤随机噪声) → 多条件融合(交叉验证)。

## 9. 后续优化方向

#### 方向 1: 接入真实 BPU 推理替换 CPU 后备

```

  2. 配置 hobot_dnn 节点参数
  3. 验证 /person_detections 话题输出

收益: 帧率 6x 提升, CPU 负载降低 80%, 功耗降低
```

```

操作:
  1. 地平线 Model Zoo 下载火焰检测模型
  2. 在 hobot_dnn 中加载第二个模型 (BPU双核并行)

```

#### 方向 3: 语音播报替代蜂鸣音效

```

```

```

## 10. 故障排查

### 10.1 摄像头不工作

```bash
dmesg | tail -30 | grep -iE "camera|video|error"
ls /dev/video*
sudo apt install v4l-utils
v4l2-ctl --list-devices
v4l2-ctl --device=/dev/video0 --all
```

### 10.2 M260C 没声音

```bash
# 检查硬件
lsusb | grep -i audio              # 设备是否存在
aplay -l                            # 声卡是否注册
cat /proc/asound/cards              # 声卡列表

# 测试播放
aplay -D plughw:1,0 /tmp/test_beep.wav 2>&1
# 常见错误:
#   "No such file" → card编号不对, 换 plughw:2,0
#   "Device busy"  → sudo fuser /dev/snd/*  找出占用的进程
#   "Broken pipe"  → 采样率/格式不匹配, 用 plughw 自动转换

# 音量
amixer -c 1 scontrols               # 查看混音器控件
amixer -c 1 set PCM 90%             # 调高音量
```

### 10.3 LED 不亮

```bash
# 检查接线: 正负极是否反了? 电阻是否串联?
# 手动测试 GPIO:
echo 23 > /sys/class/gpio/export
echo out > /sys/class/gpio/gpio23/direction
echo 1 > /sys/class/gpio/gpio23/value    # 亮
echo 0 > /sys/class/gpio/gpio23/value    # 灭
echo 23 > /sys/class/gpio/unexport

# 如果用 RPi.GPIO 报错, 换 sysfs (代码已自动fallback)
```

### 10.4 vision_monitor 不工作

```bash
ros2 topic info /person_detections           # 检查检测话题
ros2 topic info /camera/aligned_depth_to_color/image_raw  # 检查深度话题
ros2 param get /vision_monitor depth_topic   # 确认话题名匹配
ros2 topic echo /camera/color/camera_info --once  # 确认相机内参
ros2 run ucar_vision vision_monitor --ros-args --log-level debug
```
